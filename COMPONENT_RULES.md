# KBO Records Component Rules

> 기준 시안: **A. Press Box Ledger**
> 목적: 기록 조회 중심 웹에 맞는 컴포넌트 사용 규칙. 모든 컴포넌트는 숫자와 표를 더 잘 읽게 해야 한다.

---

## 1. 공통 원칙

### 1.1 데이터가 장식보다 먼저다

컴포넌트는 예쁜 박스가 아니라 기록을 읽기 위한 장치다.

- 중요한 숫자는 가장 먼저 읽혀야 한다.
- 카드, 배지, 아이콘은 데이터의 의미를 보조해야 한다.
- 넓은 장식 영역은 피한다.
- 한 화면에 보이는 기록량을 불필요하게 줄이지 않는다.

### 1.2 컴포넌트는 기록지의 부품처럼 보여야 한다

모든 UI는 다음 질감을 공유한다.

- 얇은 선
- 낮은 radius
- 절제된 그림자
- 명확한 표 헤더
- 작은 배지와 상태 칩
- 팀 컬러의 제한적 사용

## 2. Page Layout

### 2.1 기본 구조

Desktop 기본 구조:

```txt
Sidebar | Header
        | Main content + optional right rail
```

권장 폭:

- Sidebar: `220px`~`240px`
- Main max width: `1280px`~`1440px`
- Right rail: `280px`~`340px`
- Page padding: desktop `24px`, mobile `16px`

### 2.2 홈 화면

홈은 기록실 입구다.

필수:

- 전역 검색
- 시즌 팀 순위
- 주요 리더 요약
- 업데이트 시간
- 데이터 출처

선택:

- 최근 주목 선수
- 시즌별 빠른 이동
- 용어 사전 바로가기

제외 또는 낮은 우선순위:

- 경기 일정/결과
- 실시간 경기 상황
- 뉴스형 hero

### 2.3 리더보드

리더보드는 필터와 순위 결과가 한 화면에서 연결되어야 한다.

구성:

- 페이지 제목
- 타자/투수 segmented control
- 시즌 select
- 팀 select
- 지표 그룹: 기본 기록 / 세이버메트릭스
- TOP 3 요약
- 전체 순위 표

규칙:

- 필터 영역은 한 패널에 모아 둔다.
- 선택된 지표명은 표 헤더와 TOP 카드에 동일하게 반영한다.
- 하위 순위는 카드가 아니라 조밀한 표로 보여준다.

### 2.4 선수 목록

선수 목록은 스프레드시트처럼 읽히되, 너무 차갑지 않게 한다.

구성:

- 타자/투수 segmented control
- 팀 필터
- 선수명 검색
- 규정타석/규정이닝 toggle
- 기록표

규칙:

- 첫 열은 순위 또는 번호, 두 번째 열은 선수명과 팀 배지
- 숫자 열은 오른쪽 정렬
- 정렬 가능한 열은 hover와 active 상태 표시
- 모바일에서는 핵심 4~6개 열만 우선 표시하고 상세로 연결한다

### 2.5 선수 상세

선수 상세는 “프로필 카드”보다 “개인 기록 페이지”에 가깝게 만든다.

구성:

- 선수명, 팀, 포지션, 시즌
- 핵심 지표 카드
- 상세 기록표
- 연도별 통산 기록표
- 필요 시 간단한 능력 시각화

규칙:

- 사진은 있으면 좋지만 기록보다 크지 않게 둔다.
- 레이더 차트는 보조 정보로만 둔다.
- 통산 기록표는 가로 스크롤을 허용하되 첫 열은 sticky 처리한다.

### 2.6 FA 화면

FA 화면은 스카우팅 보드처럼 보일 수 있지만, 시안 A에서는 과한 질감을 줄이고 기록표 질서를 유지한다.

구성:

- 예상 인원, 등급 분포, 예상 시장 규모, 보상 대상 요약
- 포지션/팀/등급 필터
- FA 선수 표
- 클릭 시 상세 분석

규칙:

- 등급 배지는 작고 명확하게
- 예상 계약 규모는 오른쪽 정렬
- 설명형 문구는 상세 모달이나 우측 레일로 분리

## 3. Sidebar

### 3.1 역할

사이드바는 조용한 목차다. 브랜드와 메뉴를 보여주되 본문보다 강하면 안 된다.

권장 메뉴:

- 홈
- 리더보드
- 팀
- 선수
- 2027 FA
- 용어 사전

경기 일정/결과 메뉴는 기본 메뉴에서 제외한다.

### 3.2 상태

| 상태     | 규칙                                            |
| -------- | ----------------------------------------------- |
| Default  | 배경 없음 또는 아주 옅은 hover 가능             |
| Hover    | `accent` 배경, 텍스트 선명도 증가               |
| Active   | 왼쪽 2px 선 + 옅은 배경                         |
| Disabled | 사용하지 않는다. 준비 중 메뉴는 노출하지 않는다 |

### 3.3 구성

메뉴 항목:

- 아이콘 `16px`~`20px`
- 라벨 `14px`
- 보조 설명 `11px` 또는 생략
- 높이 `44px`~`52px`

## 4. Header & Search

### 4.1 Header

헤더는 기록 조회를 위한 도구막대다.

포함:

- 전역 검색
- 업데이트 시간
- 테마 전환

비권장:

- 큰 로고 반복
- 페이지 설명 문구
- 마케팅 CTA

### 4.2 Search

전역 검색은 최상위 기능이다.

규칙:

- placeholder: `선수명, 팀명으로 검색...`
- 입력 높이: `36px`~`40px`
- 결과는 최대 8개 표시
- 결과 항목에는 선수명, 팀명, 타자/투수 구분을 표시
- 키보드 탐색을 지원한다

---

## 5. Tables

### 5.1 기본 표

표는 이 서비스의 핵심 컴포넌트다.

규칙:

- 헤더 배경은 `muted`
- 행 구분은 `1px border`
- 행 높이는 `36px`~`44px`
- 숫자 열은 오른쪽 정렬
- 선수/팀명 열은 왼쪽 정렬
- 첫 열은 순위 또는 번호
- 정렬 가능한 헤더에는 작은 방향 아이콘을 둔다

### 5.2 Density

기록표는 너무 넓게 만들지 않는다.

권장:

- `py-2` 또는 `py-2.5`
- `px-3`
- 폰트 `13px`~`14px`

비권장:

- `rounded-2xl` 표 컨테이너 남발
- 행 높이 56px 이상
- 각 행을 카드처럼 분리

### 5.3 Sticky

가로 스크롤 표에서는 다음을 권장한다.

- 첫 번째 선수/팀 열 sticky
- 헤더 sticky는 페이지 길이가 긴 표에서만 사용
- sticky 배경은 표 배경과 반드시 일치

### 5.4 Table Footer Note

표 하단 주석은 짧고 정확하게 쓴다.

예:

- `승률이 같은 경우: 승수 → 상대 전적 → 득실차 → 추첨 순으로 순위를 결정합니다.`
- `규정타석 충족 기준 적용`
- `데이터 출처: KBO 공식 사이트`

## 6. Cards & Panels

### 6.1 Panel

큰 정보 영역은 card보다 panel로 본다.

규칙:

- 배경 `card`
- border `line-hair`
- radius `6px`
- shadow 거의 없음
- 제목 영역과 본문 영역을 선으로 구분

사용처:

- 팀 순위 패널
- 필터 패널
- 상세 기록 영역
- FA 요약 영역

### 6.2 Summary Card

요약 카드는 숫자 하나와 짧은 맥락을 보여준다.

구성:

- label
- main value
- unit
- subject
- optional team badge/photo

규칙:

- 큰 숫자는 `font-stat`
- 카드 안 이미지는 보조 크기로 제한
- 배경 이미지는 쓰지 않는다
- team color는 상단선 또는 작은 border에만 사용

### 6.3 Top Rank Card

리더보드 TOP 3 전용 카드.

규칙:

- 1위만 약간 더 강하게 표현 가능
- 선수 사진은 `56px`~`72px`
- 수치가 사진보다 먼저 읽혀야 한다
- 2위/3위는 같은 구조를 유지한다

---

## 7. Filters

### 7.1 Segmented Control

타자/투수, 기본/세이버 같은 상호 배타 선택에 사용한다.

규칙:

- 높이 `36px`
- radius `4px`
- active 상태는 배경보다 border/텍스트 대비로 표현
- 항목 수는 2~4개 권장

### 7.2 Select

시즌, 팀, 등급처럼 목록 선택에 사용한다.

규칙:

- 값은 짧게: `2026시즌`, `전체`, `KIA`
- trigger 폭은 내용에 맞추되 흔들리지 않게 고정
- 빈 value는 사용하지 않는다

### 7.3 Toggle

규정타석/규정이닝처럼 on/off 조건에 사용한다.

문구 예:

- `규정타석`
- `규정이닝`
- `전체 포함`

ON/OFF 문구를 길게 쓰지 않는다.

---

## 8. Badges & Chips

### 8.1 Team Badge

팀 배지는 작은 식별자다.

규칙:

- 로고가 있으면 로고 우선
- 로고가 없으면 2~3글자 약칭
- 팀 컬러 border 또는 작은 dot 사용
- 큰 배경색으로 채우지 않는다

### 8.2 Result Chip

최근 10경기 결과에 사용한다.

규칙:

- `W`, `L`, `D` 문자 필수
- 색상만으로 의미 전달 금지
- 같은 크기와 간격 유지
- 오른쪽이 최신인지, 왼쪽이 최신인지 한 곳에서 규칙화한다

### 8.3 Rank Badge

순위는 badge보다 숫자 자체를 명확하게 둔다.

권장:

- 1~3위만 약한 강조
- 나머지는 같은 톤 유지

비권장:

- 모든 순위에 큰 원형 badge
- 금/은/동 과장 효과

---

## 9. Buttons & Links

### 9.1 Button

버튼은 명령에만 사용한다.

권장:

- `전체 보기`
- `새로고침`
- `기록 보기`
- `비교 추가`

비권장:

- 설명 문장형 버튼
- 장식용 CTA
- 과한 pill button 남발

### 9.2 Icon Button

아이콘만 있는 버튼은 다음 조건을 만족해야 한다.

- 익숙한 아이콘일 것
- `aria-label` 제공
- tooltip 제공
- 크기 `32px`~`36px`

### 9.3 Links

기록 상세로 이동하는 텍스트는 링크처럼 보여야 한다.

규칙:

- 선수명/팀명 hover 시 underline 또는 primary color
- 표 안 링크는 행 전체 클릭과 충돌하지 않게 처리

---

## 10. Dialogs

상세 분석이나 보상 규정처럼 긴 내용을 보여줄 때 사용한다.

규칙:

- 제목은 짧게
- 본문은 섹션별로 나눈다
- 숫자는 표 또는 key-value 형태로 정리
- 닫기 버튼과 ESC 동작 보장
- 모바일에서는 sheet/drawer 형태를 고려

---

## 11. Empty, Loading, Error

### 11.1 Loading

기록 사이트의 loading은 차분해야 한다.

권장:

- 표 skeleton
- 얇은 상단 loading bar
- 셀 단위 placeholder

비권장:

- 큰 spinner 중앙 배치 남발
- 과한 shimmer

### 11.2 Empty

검색 결과 없음은 짧게 안내한다.

예:

- `검색 결과가 없습니다.`
- `조건에 맞는 선수가 없습니다.`

다음 행동이 명확하면 함께 제공한다.

- `필터 초기화`
- `전체 보기`

### 11.3 Error

오류는 사용자에게 원인을 쉽게 말한다.

예:

- `기록을 불러오지 못했습니다. 잠시 후 다시 시도해 주세요.`
- `KBO 공식 사이트 응답이 지연되고 있습니다.`

개발자용 stack trace는 화면에 노출하지 않는다.

## 12. Mobile Rules

모바일에서 모든 표 열을 억지로 유지하지 않는다.

### 12.1 우선순위

선수 목록 타자:

1. 선수
2. 팀
3. AVG
4. HR
5. RBI
6. OPS

투수:

1. 선수
2. 팀
3. ERA
4. W
5. SO
6. WHIP

팀 순위:

1. 순위
2. 팀
3. 승-패-무
4. 승률
5. 게임차

### 12.2 모바일 패턴

- 표는 가로 스크롤 허용
- 중요한 첫 열은 sticky
- 필터는 2줄까지 허용
- 오른쪽 레일은 본문 아래로 이동
- 사이드바는 drawer로 전환

## 13. Do / Don't

### Do

- 표와 숫자를 가장 먼저 설계한다.
- 팀 컬러는 작은 신호로 사용한다.
- 기록 출처와 업데이트 시간을 남긴다.
- 긴 설명은 tooltip, 주석, 상세 영역으로 분리한다.
- 화면마다 같은 필터 구조를 유지한다.

### Don't

- 경기 일정/결과를 주요 기능처럼 강조하지 않는다.
- 페이지를 큰 카드들의 모음으로만 만들지 않는다.
- 무의미한 스포츠 이미지를 배경에 깔지 않는다.
- 그라디언트와 네온으로 정보를 꾸미지 않는다.
- 텍스트가 버튼이나 표 셀 안에서 줄바꿈으로 깨지게 두지 않는다.

## 14. Implementation Checklist

화면 구현 전 확인:

- [ ] 이 화면의 핵심 기록은 무엇인가?
- [ ] 사용자가 가장 먼저 볼 숫자는 무엇인가?
- [ ] 필터는 한눈에 이해되는가?
- [ ] 표의 숫자 정렬과 소수점 자릿수는 일관적인가?
- [ ] 팀 컬러가 정보 식별 이상의 면적을 차지하지 않는가?
- [ ] 모바일에서 핵심 열이 살아남는가?
- [ ] 로딩/빈 상태/오류 상태가 준비되어 있는가?
- [ ] 시안 A의 기록지 느낌을 해치지 않는가?

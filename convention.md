# 개발 컨벤션 & 기초 프로세스 정리

> 바이브 코딩에서 실무형 협업 습관으로 넘어가기 위한 기초 가이드.
> 지금까지 살펴본 내용(Git 협업 흐름, 브랜치, 커밋 컨벤션 등)을 한 곳에 정리한 문서.

---

## 1. Git 협업 기본 흐름

혼자 `main`에 바로 커밋·푸시하는 방식과 달리, 회사에서는 **브랜치 + PR + 리뷰** 단계를 거친다.
핵심 원칙: **`main` 브랜치는 항상 "배포 가능한 안정 상태"로 유지**하고, 작업은 별도 브랜치에서 한다.

### 전체 흐름

```
main 최신화  →  작업 브랜치 생성  →  코드 작성 + 커밋  →  푸시  →  PR 생성  →  리뷰 + CI  →  머지  →  브랜치 삭제 + 배포
```

### 단계별 명령어

```bash
# 1. main을 최신으로 맞추기
git checkout main
git pull origin main

# 2. 작업용 브랜치 생성 (feature/기능 또는 이름/기능 형태)
git checkout -b feature/login-page

# 3. 작업하면서 의미 단위로 커밋
git add .
git commit -m "feat: 로그인 폼 UI 추가"

# 4. 원격 저장소에 브랜치 푸시
git push origin feature/login-page

# 5. GitHub에서 PR(Pull Request) 생성
#    feature/login-page -> main 으로 합치겠다는 "요청서"

# 6. 리뷰 + CI 자동검사 통과

# 7. 머지(Merge) 후 브랜치 삭제
```

---

## 2. 브랜치란?

- 브랜치 = **코드의 독립된 작업 줄기**(평행우주). `main`에서 가지를 쳐서 따로 작업한다.
- 그 줄기에서 마음껏 커밋·실험해도 **`main`은 영향받지 않는다.**
- 작업이 끝나면 다시 `main`에 합친다(머지).

### 로컬 vs 원격

| 구분 | 보이는 범위 |
|------|------------|
| **로컬 브랜치** (`git checkout -b`만 한 상태) | 내 컴퓨터에만 존재. 아직 아무도 못 봄 |
| **원격 브랜치** (`git push` 후) | GitHub에 올라가 **팀 전원이 볼 수 있음** |

> 브랜치를 쓰는 이유: ① 격리(내 작업이 끝날 때까지 main을 안 건드림) ② 병렬 작업 ③ 검토 단위(브랜치 1개 = PR 1개)

---

## 3. PR(Pull Request)이란?

- 코드를 바로 `main`에 넣지 않고 **"이거 합쳐도 될까요?" 하고 올리는 요청서**.
- 리뷰어를 지정하고, 무엇을·왜 바꿨는지 설명을 적는다.
- 검토 두 가지: **사람(동료 승인)** + **기계(CI 자동검사: 빌드/테스트/린트)**.
- 수정 요청이 오면 같은 브랜치에서 다시 커밋·푸시하면 PR에 자동 반영된다.

## 4. 커밋 메시지 컨벤션 (Conventional Commits)

커밋이 "무슨 종류의 작업인지" 앞에 타입 딱지를 붙이는 업계 표준 규칙.

### 형식

```
타입: 무엇을 했는지 설명
```

### 주요 타입

| 타입 | 의미 | 언제 쓰나 | 예시 |
|------|------|----------|------|
| `feat` | 새 기능 | 기능을 추가했을 때 | `feat: 좋아요 버튼 추가` |
| `fix` | 버그 수정 | 버그·오류를 고쳤을 때 | `fix: 댓글 안 보이던 문제 수정` |
| `chore` | 잡일 | 기능과 무관한 설정·정리 | `chore: 불필요한 로그 삭제` |
| `refactor` | 리팩터링 | 동작은 그대로, 코드만 정리 | `refactor: 함수 구조 개선` |
| `docs` | 문서 | README 등 문서 수정 | `docs: 설치 방법 추가` |
| `style` | 스타일 | 들여쓰기·세미콜론 등 겉모양만 | `style: 코드 포맷 정리` |
| `test` | 테스트 | 테스트 코드 추가/수정 | `test: 로그인 테스트 추가` |
| `perf` | 성능 | 성능 개선 | `perf: 이미지 로딩 속도 개선` |

> 가장 많이 쓰는 건 `feat`(새 기능)와 `fix`(버그 수정). 처음엔 이 둘만 구분해도 충분하다.

### 헷갈리기 쉬운 구분

- **`fix` vs `refactor`**: `fix`는 잘못 동작하던 걸 고침(결과가 바뀜) / `refactor`는 잘 되던 코드를 더 깔끔하게(결과 그대로)
- **`refactor` vs `style`**: `refactor`는 구조·로직 정리 / `style`은 띄어쓰기·줄바꿈 등 겉모양만

### 적용 예시

```bash
# 전 (자유 형식)
git commit -m "버튼 추가함"
git commit -m "오류 고침"

# 후 (컨벤션 적용)
git commit -m "feat: 다운로드 버튼 추가"
git commit -m "fix: 로그인 시 멈추던 오류 수정"
```

---

## 5. 브랜치 이름 컨벤션

커밋 타입과 비슷하게 브랜치 이름도 용도를 앞에 붙인다.

| 형태 | 용도 | 예시 |
|------|------|------|
| `feat/...` | 새 기능 | `feat/mobile-view` |
| `fix/...` | 버그 수정 | `fix/sort` |
| `refactor/...` | 리팩터링 | `refactor/routing` |

---

## 6. 따라 하면 좋은 기초 습관 체크리스트

- [ ] `main`에 직접 커밋하지 말고 **브랜치 따서 작업** → PR 올려보기
- [ ] 커밋은 **의미 단위로 작게** 쪼개기 (한 커밋 = 한 가지 변경)
- [ ] 커밋 메시지에 **`feat:` / `fix:` 타입** 붙이기
- [ ] 작업 시작 전 **`git pull`로 main 최신화** 습관
- [ ] 브랜치 이름도 **`feat/` `fix/`** 규칙 따르기
- [ ] (배포 시) PR 미리보기 배포로 **합치기 전에 화면 확인**

> 처음부터 전부 지키려 하지 말고, **브랜치 만들기 + `feat`/`fix` 커밋 메시지** 두 가지부터 시작하면 충분하다.
